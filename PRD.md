# PRD — 二十八星宿测算 Web App（本命宿 · 值日法）

> 状态：DRAFT 0.3 · 2026-07-12（5 项已定 + 后端定为同仓 Vercel Functions，见 §8/§15）
> 定位：这份 PRD 是本项目「产品的唯一真相源（Source of Truth）」—— 人和 AI 动手前都回这份读、这份改。
> 平台：响应式 Web App（React + Vite 前端 + 同一 Vercel 项目的 `api/` Serverless Functions 当后端），手机浏览器可用。
> 一句话：**输入公历或农历生日（年月日时）→ 算出你的本命星宿 → 看它的形象、星图与多维度解读。**

---

## 1. 目标（这个产品要解决什么）

让普通用户不用懂任何历法，只输入自己的生日，就能：

1. 知道自己的**本命星宿**（二十八宿之一）；
2. 看到这个星宿的**形象与星图**（视觉）；
3. 读到这个星宿的**多维度解读**：意义、本命、一生命运、性格、事业、财运、爱情。

并且登录后能**保存/回看**自己（及家人朋友）的测算结果。

> 心态定位：文化 + 娱乐向的自我认知工具，不是命理预测服务。红线见 §12。

---

## 2. 目标用户

- 对传统文化 / 星座命理 / 自我探索感兴趣的大众用户（年轻人为主）。
- 不懂农历换算、不懂二十八宿，只想「输入生日，看结果」。
- 会分享结果到社交媒体（截图 / 分享卡片是重要传播点）。

---

## 3. MVP 边界（先做什么，先不做什么）

### 核心动作（MVP 只保证这一个 Flow 顺畅）
> **输入生日 → 得到本命宿 → 看解读**。其余都围绕它。

### MVP 做
- 生日输入（公历/农历切换，年月日 + 时辰）。
- 本命宿测算（值日法，见 §7）。
- 结果页：星宿形象 + 星图 + 七个解读板块 + 时辰辅助板块（选了时辰才出）。
- 邮箱直登（登录 = 注册合一，见 §8）+ 保存我的测算记录、回看历史。
- 一份可分享的结果卡片（截图即可，不做复杂分享 SDK）。

### MVP 不做（后续）
- 付费 / 会员 / 解锁高级报告。
- 合盘（两人星宿配对）、流年运势、每日运势。
- AI 对话式解读、语音。
- 多语言（先中文）。
- 复杂社交（关注、社区、评论）。

## 4. 核心 Flow（主流程）

```
落地页
  → [公历 / 农历] 切换
  → 选 年 / 月 / 日 / 时辰（时辰可选“不确定”）
  → 点「测算」
  → 载入动画（排盘感）
  → 结果页：
       顶部：本命宿名（如「房日兔」）+ 所属四象（东方青龙）+ 星图/形象
       中部：七个解读板块（意义 / 本命 / 一生命运 / 性格 / 事业 / 财运 / 爱情）
       条件：时辰辅助板块（仅当填了时辰时显示）
       底部：保存（需登录）/ 生成分享卡片 / 再测一个
```

- 未登录也能测算和看结果；**保存/历史**才要求登录（降低首次体验门槛）。
- 时辰选「不确定」：本命宿照常算（值日宿按“日”定），时辰辅助板块隐藏。

## 7. 测算模块（🔑 核心引擎：中国二十八宿值日法）

### 方法
**二十八宿值日法**：以固定的值日顺序，每日轮值一宿，二十八日一循环、连续不断地纪日。**本命宿 = 出生当日的值日宿。**

- 二十八宿（四象 × 七宿）：
  - 东方青龙：角 亢 氐 房 心 尾 箕
  - 北方玄武：斗 牛 女 虚 危 室 壁
  - 西方白虎：奎 娄 胃 昴 毕 觜 参
  - 南方朱雀：井 鬼 柳 星 张 翼 轸
- 每宿附带传统属性（七政/七曜 + 动物），如「房日兔」「心月狐」—— 由库直接给出（见下）。

### 输入 → 计算
1. 用户输入公历或农历「年月日时」。
2. 用历法库把输入统一成日期对象（农历 / 公历都支持）。
3. 取该日的**值日宿** = 本命宿；同时取七政、动物（房 + 日 + 兔 = 房日兔）。
4. **时辰 → 辅助解读（决策 3）**：用出生时辰（地支时）驱动一个「时辰辅助」板块（时辰对性格的微调 / 宜忌等），内容取自释义库；选「不确定」则隐藏该板块，本命宿照常算。

### 🚨 实现方式（已定：用成熟历法库，不手搓 — 决策 1）
- **选定库：`lunar-javascript`（6tail，MIT，多语言同源移植，社区成熟）** —— 它**同时**提供公历↔农历换算 **和** 值日二十八宿：
  - `Solar.fromYmd(y,m,d).getLunar()`（或 `Lunar.fromYmd(...)`）→
    `lunar.getXiu()`（宿）、`getZheng()`（七政）、`getAnimal()`（动物）、`getXiuLuck()`（吉凶）；时辰用 `getTimeZhi()` 等。
- 因此**不手写值日序 / 锚定日，也不手写农历换算表** —— 由库负责，规避「写死一张错表」的风险。
- 仍要**交叉校验**：上线前用 **≥3 个已知本命宿的样本**核对库输出（不臆造，眼见为实）。
- 测算封装成**可替换模块** `computeBenmingXiu(input) → { xiu, zheng, animal, siXiang, timeZhi? }`，换库 / 换体系不影响页面层。

## 8. 登录 & 后端

- **目的**：邮箱登录 + 保存/回看测算记录（生日 + 结果），后续做个性化。
- **登录（已定：邮箱免验证直登 — 决策 2）**：只填邮箱即视为登录，**不发验证码 / magic link，也不自建密码体系**。
  - ⚠️ 邮箱在此**仅作软标识**（未验证所有权，任何人可填任意邮箱）——这是 MVP 降低体验门槛的**刻意取舍**；因此生日等敏感数据的保护以「未登录不强制留存 + 用户可删除」为准（见 §12），不把「已登录」当成身份可信。
- **注册 = 与登录合一（无独立注册步骤）**：免验证直登下**不区分「注册」和「登录」**——首次填入某邮箱即**自动建号**，之后再填同一邮箱即**登录同一账号**。前端只有 `authAdapter.login(email)` 一个动作兼顾建号 / 登录（后端对该邮箱做 upsert）；不存在单独的注册页 / 注册表单。
- **后端 = 同一个 Vercel 项目里的 `api/` Serverless Functions**（跟前端同仓、同一次部署、同一个域名，不另起服务）：
  - Functions 提供 API 端点：保存 / 读取 `Reading`、按邮箱维持会话等。
  - **会话 & 存储**用成熟方案对接（Supabase 或 Vercel Postgres/KV，见 §15）；免验证直登**不需要邮件通道（OTP / magic link）**，Functions 只负责登记邮箱、维持会话、读写记录 —— 仍是 API 层，**不从零手搓鉴权**。
  - **数据存储**：Vercel Postgres / KV，或 Supabase Postgres（二选一，见 §15）。
- **前端只依赖两个接口**：`authAdapter`（login / logout / currentUser）+ `readingStore`（save / list）—— 后端实现（Functions + auth + 存储）可整体替换，不影响页面。
- **配合 L4**：scaffold 阶段这两个接口先做**占位**（本地存储 / mock）；真后端（Vercel Functions + auth + DB）放到 Vercel 上，作为**下一步动态能力**接入 —— 正好是 L4 收尾说的「动态能力后续在 Vercel 扩展」。
- **账号数据最小化**：只存必要标识（邮箱）+ 用户主动保存的记录；生日属敏感信息（见 §12 隐私红线）。

## 9. 数据模型 Data（先 mock / fixture，后接真实）

```
User        { id, email, createdAt }
XiuEntry    { id(宿名), 名称, 四象, 七政, 动物,
              formImage, starMap,
              意义, 本命, 一生命运, 性格, 事业, 财运, 爱情,
              时辰辅助?: { [地支时]: 文案 } }              // = 释义库一条
Reading     { id, userId?, 输入{历法, 年,月,日,时辰}, 公历date, benmingXiuId, createdAt }
```

- `XiuEntry` × 28 条 = **释义内容库**（§10），静态数据，前端可直接读 JSON。
- 七政 / 动物既可由库实时算，也可冗余进 `XiuEntry` 便于展示；以库为准。
- `Reading` = 一次测算结果；登录后与 `userId` 绑定进入 `History`。
- MVP 阶段：`XiuEntry` 用真实整理好的内容 JSON；`User`/`Reading` 走 `authAdapter` / `readingStore` 接口（真实现 = 同仓 Vercel Functions + 存储；登录前可临时存本地）。

## 10. 内容资产（释义库 + 图）

- **28 宿 × 九类内容**（形象、星图、意义、本命、一生命运、性格、事业、财运、爱情）+ 时辰辅助文案 = 结构化 JSON（每宿一条 `XiuEntry`）。
- **MVP 上全真内容（决策 4）**：28 宿全部按真实传统资料整理齐，**不留占位**。这是 MVP 的一项硬内容工作量，需排期。
- **🚨 内容纪律**：解读文案**基于可查的传统资料整理、来源可追溯，不由 AI 现编冒充权威**。可用 AI 辅助起草，但必须核对传统出处、标注来源、人工过一遍定稿。
- **图片：AI 模型生成（决策 5）**：形象图 / 星图由生图模型产出，统一风格（东方 + 星空），28 宿一致。
  - 版权：生成图作为原创素材使用；选用**允许商用**的模型 / 授权，记录所用模型与生成参数（可追溯）。
  - ⚠️ **两类图要分清**：①**星宿形象图**（艺术形象）——AI 生成没问题、正是它擅长的；②**星图**——若要表达**真实星官位置**（天文准确），生图模型会画错、不可信；本项目定位是文化娱乐，**星图按「装饰性 / 示意」处理，不宣称天文精确**（若日后要精确星官，另基于真实星表坐标绘制）。

## 11. 设计 / 视觉 SoT（tokens）

- 复用「视觉真相源」`tokens.css`：颜色 / 字体 / 间距 / 圆角 / 阴影统一变量化。
- 风格方向：东方 + 星空感（四象四色可作分类色：青龙-青、朱雀-红、白虎-白/金、玄武-黑/蓝）。**具体色板在 tokens 里定，页面只引变量。**
- AI 生成的形象图 / 星图与 tokens 色系要协调（生成时把四象色带进 prompt，保持整套视觉统一）。
- 结果卡片要考虑「适合截图分享」的构图。

## 12. 红线（绝不违反）

1. **文化娱乐定位**：结果页与 About 页必须有**显著免责声明** —— 内容为传统文化 / 娱乐参考，不构成人生、医疗、投资、婚姻等任何决策建议。
2. **不臆造命理内容**：解读文案基于可查资料整理、标来源，不 AI 现编冒充权威（§10）；测算用成熟历法库并交叉校验，不写死可能错误的表（§7）。
3. **隐私**：生日 + 邮箱是敏感个人信息 —— 明确告知用途、未登录不强制留存、登录用户可删除自己的记录与账号；不向第三方售卖。
4. **不制造焦虑 / 不诱导付费迷信**：不用「命中注定破财 / 必须化解」这类恐吓式文案（MVP 也不含付费）。
5. **合规**：AI 生成图选允许商用的模型 / 授权并记录来源，不盗用他人素材。
